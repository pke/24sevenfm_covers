Code signing policy

Applies from 17 July 2026

How the downloads on this site are built, approved and signed — and how anyone can check that for themselves.

Status: not yet signed

An application to the SignPath Foundation is pending. Until it is granted, the downloads are not signed, and Windows SmartScreen will warn the first time an installer runs. That warning is expected, and the policy below describes how signing will work once the certificate is issued — it does not describe the current downloads.

Signing

Free code signing provided by SignPath.io, certificate by SignPath Foundation.

Windows will name “SignPath Foundation” as the verified publisher — not DudeSoft, and not this project. That is correct and expected, not a sign that something is wrong: the certificate is issued to the SignPath Foundation, which is therefore the publisher of record. This project is identified instead by the product name 24seven.fm Covers in each binary's file metadata.

What the signature actually attests to is the useful part: that the binary is a valid, automated build produced from the source code in this project's repository — not merely that somebody, somewhere, signed it.

Team roles

This is a single-maintainer project, so one person holds every role. Naming that plainly is more honest than implying a review board that does not exist:

  • Author Philipp Kursawe (@pke) — writes the code and triggers the release build.
  • Reviewer Philipp Kursawe (@pke) — reviews what goes into a release.
  • Approver Philipp Kursawe (@pke) — approves each signing request individually. No release is signed automatically.

Multi-factor authentication is enabled for both source-repository and signing-service access.

How a release is built

Every artifact is built from the source in this repository by a public GitHub Actions workflow — never on a developer's machine, and never uploaded by hand:

  • Source github.com/pke/24sevenfm_covers, MIT licensed. Every dependency is vendored into the repository as source, so a build pulls in nothing that isn't visible there.
  • Build The Release workflow runs the unit-test gate, compiles all three binaries, and publishes them as a GitHub release together with a .sha256 sidecar for every file.
  • Verify Compare a download against its .sha256 sidecar on the release page, and the workflow log shows exactly which commit produced it.

Privacy

The plugins and the standalone player collect no data, and this site sets no cookies and runs no analytics. The details, including every file written to your own machine, are in the privacy policy.
